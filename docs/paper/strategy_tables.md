# QQN Strategy Tables

The tables below enumerate the concrete strategies available on each of QQN's four orthogonal axes (oracle, path, line
search, region), plus the cross-cutting combinators and enhancements. Each row lists the strategy name, a brief
description of its properties, its computational cost, and the problem domains where it helps or hurts.

---

## 1. Oracle Strategies

The oracle supplies the `t = 1` endpoint `-H∇f` of the quadratic path. Because the path always retains the gradient
tangent at the origin, the oracle need not guarantee descent on its own.

| Name        | Description / Properties                                                                                                  | Computational Cost                                                        | Problem Domain Pros / Cons                                                                                                                                                                   |
|-------------|---------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Gradient    | Trivial oracle returning `-∇f`; path degenerates to scaled steepest descent. Essential early-iteration/terminal fallback. | `O(n)`; negligible (no state)                                             | **Pro:** always a descent direction, robust, memory-free. **Con:** slow on ill-conditioned/anisotropic problems.                                                                             |
| Momentum    | Heavy-ball direction integrating decaying history of parameter deltas `Δx`. One hyperparameter `β` (default 0.9).         | `O(n)` per step; `O(n)` state (velocity `v`)                              | **Pro:** smooths path, damps oscillation, cheap. **Con:** no true curvature; weaker than quasi-Newton on anisotropic Hessians.                                                               |
| Adam        | Adaptive moment estimation: decaying first moment `m` + second moment `v` with bias correction. `β₁, β₂, ε` defaults.     | `O(n)` per step; `O(n)` state (two moment buffers)                        | **Pro:** per-coordinate rescaling, robust default in ML. **Con:** diagonal-only preconditioning; misses cross-coordinate curvature.                                                          |
| L-BFGS      | Default oracle. Quasi-Newton `-H∇f` via two-loop recursion over `m` curvature pairs `(s, y)`; curvature-safeguarded.      | `O(m·n)` per step; `O(m·n)` memory (`m` = history size)                   | **Pro:** captures dominant curvature subspace; superlinear near optimum; wins the empirical benchmarks. **Con:** memory scales with `m`; history can stale/degenerate on non-convex terrain. |
| Secant (BB) | Matrix-free Barzilai-Borwein step; scalar step estimate from a single secant pair.                                        | `O(n)` per step; `O(n)` memory                                            | **Pro:** curvature-aware yet matrix-free; a robust backup carrying curvature a momentum fallback lacks. **Con:** coarse scalar curvature vs. full L-BFGS subspace.                           |
| Shampoo     | Structure-aware preconditioner using Kronecker-factored second-moment statistics per tensor.                              | Superlinear in subspace dim (factor construction/inversion); larger state | **Pro:** exploits tensor structure; strong on structured weights. **Con:** expensive per step unless partitioned; heavier memory.                                                            |
| Anderson    | Anderson/Pulay acceleration — the variational ideal that L-BFGS approximates.                                             | `O(k·n)` per step (`k` = window); `O(k·n)` memory                         | **Pro:** principled acceleration from residual history. **Con:** cannot match deep L-BFGS subspace capture on anisotropic Hessians.                                                          |

### 1.1 Oracle Combinators & Structural Variants

| Name            | Description / Properties                                                                                                      | Computational Cost                                                            | Problem Domain Pros / Cons                                                                                                                                     |
|-----------------|-------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Fallback([...]) | Uses first *descent-valid* child direction, else falls through; terminal safety net returns `-∇f`. Hyperparameter-free.       | Evaluates all children every step (both `where` branches computed)            | **Pro:** primary robustness mechanism; deep oracle safeguarded by cheap backup. **Con:** pays for unused child directions (usually cheap vs. objective evals). |
| Blend([...])    | Fixed convex combination of multiple oracle directions.                                                                       | Sum of children's per-step cost                                               | **Pro:** static hedge across oracle types. **Con:** fixed weights lack adaptivity; still evaluates all children.                                               |
| Partitioned     | Partitions parameters (by layer/tensor/block); independent oracle regime per partition, concatenated at `t = 1`.              | Sum of per-partition costs; confines superlinear-cost oracles to small blocks | **Pro:** makes Shampoo/dense-secant far faster; per-layer curvature captured more faithfully. **Con:** ignores cross-partition coupling.                       |
| Probe-feeding   | Orthogonal enhancement: feeds line-search probe gradients into oracle memory; descent-gated and count-capped. Off by default. | Extra buffer updates; capped replay count                                     | **Pro:** reuses discarded gradient info. **Con:** collinear probes only re-estimate along `d`; over-replay flushes cross-iteration curvature.                  |

## 2. Path Strategies

The path is the geometric object the line search traverses over `t ∈ [0, 1]`.

| Name      | Description / Properties                                                                                                      | Computational Cost                                                                                                                                                                           | Problem Domain Pros / Cons                                                                                                                                                                                                            |
|-----------|-------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Linear    | Straight chord from origin to oracle endpoint; samples value-only, discards gradient. Keeps lowest feasible sample.           | `num_samples` value evals; no gradient reuse                                                                                                                                                 | **Pro:** simple control/baseline measuring value of gradient info; recovers sensible step even with degenerate direction. **Con:** discards gradient information deliberately; weakest model.                                         |
| Quadratic | Default. `d(t) = -t(1-t)∇f - t²H∇f`; begins tangent to steepest descent, ends at oracle. Anchors global convergence.          | Path construction `O(n)`; line-search probes                                                                                                                                                 | **Pro:** simplest gradient-following path; guaranteed descent at origin; superlinear near optimum; rescaling-invariant geometry. **Con:** uses only origin slope, discarding oracle-point gradient.                                   |
| Spline    | Piecewise cubic Hermite over `t`; reuses value+slope at every probe as a control point; upstream/downstream tangent symmetry. | Path construction `O(n)`; evaluating the path at an unmeasured `t` is a trivial cubic (vs. quadratic) polynomial evaluation — the same increment over quadratic as quadratic has over linear | **Pro:** reuses all measured gradients; richer curve; cheaper backtracking. **Con:** tangent reflection is a heuristic (safety inherited from inner sufficient-decrease test); incremental per-`t` cost is a near-trivial cubic term. |

## 3. Line Search Strategies

The line search traverses the path, selects `t`, enforces descent, exploits curvature, and navigates the feasible
(projected) path.

| Name                  | Description / Properties                                                              | Computational Cost           | Problem Domain Pros / Cons                                                                                                                      |
|-----------------------|---------------------------------------------------------------------------------------|------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------|
| armijo / backtracking | Robust default. Shrinks `α` until Armijo sufficient decrease holds; `lax.while_loop`. | ~1.0–1.1 evals/iteration     | **Pro:** robust, cheap, simple; JIT/vmap compatible. **Con:** no curvature condition — may leave L-BFGS pairs less well-conditioned than Wolfe. |
| strong_wolfe          | Optax zoom line search enforcing Armijo + strong curvature condition.                 | ~2.1 evals/iteration         | **Pro:** keeps L-BFGS updates well-conditioned. **Con:** can over-restrict the quadratic-path step; costlier — not the default.                 |
| hager_zhang           | Optax backtracking implementing an approximate-Wolfe scheme.                          | Moderate (approximate-Wolfe) | **Pro:** robust approximate-Wolfe balance of decrease and curvature. **Con:** more machinery than bare Armijo.                                  |
| fixed                 | Constant step size; no acceptance condition.                                          | 1 eval/iteration             | **Pro:** debugging/benchmarking baseline; deterministic. **Con:** no descent guarantee; not for real use.                                       |
| null                  | Unconditionally accepts the `t = 1` oracle point.                                     | 1 eval/iteration             | **Pro:** pure oracle-endpoint step (recovers raw L-BFGS/Newton corner). **Con:** no line-search safety; no globalization.                       |

### 3.1 Line-Search Wrappers & Enhancements

| Name              | Description / Properties                                                                                                                   | Computational Cost                                                                            | Problem Domain Pros / Cons                                                                                                                                    |
|-------------------|--------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------|
| spline_wrap       | Foreach wrapper: runs inner search, then probes spline stationary points; accepts only strict improvements.                                | Inner search; per-`t` path evaluation is a near-trivial cubic (vs. quadratic) polynomial term | **Pro:** richer path model reusing gradients; inherits inner descent guarantee. **Con:** incremental cost is the near-trivial cubic path-evaluation term.     |
| linear_wrap       | Foreach wrapper sampling the chord value-only after inner search; a deliberate control.                                                    | Inner search + `num_samples` value evals                                                      | **Pro:** baseline isolating value-of-gradient-info. **Con:** discards gradient information by design.                                                         |
| Temperature       | Metropolis-style stochastic acceptance layered on Armijo; accepts uphill with `exp(−ΔE/T)`; geometric cooling. Default `T = 0` (disabled). | Negligible (one PRNG draw per test)                                                           | **Pro:** controlled exploration/early acceptance on non-convex, multi-modal landscapes. **Con:** uphill moves may waste evals; needs cooling schedule tuning. |
| Recursive descent | Parallelizes path-building with multiple blend parameters; inner multidim QQN solves the low-dim subproblem.                               | Inner QQN over a tiny space (cheap, dense curvature)                                          | **Pro:** discovers multi-parameter blends no single scalar `t` expresses; dense accurate inner curvature. **Con:** nested-solver overhead; added complexity.  |

## 4. Region Strategies

A projective region remaps proposed updates onto a feasible/preferred set *inside* the line-search loop, so
descent/Wolfe guarantees hold on the projected path `d_R(t)`.

| Name            | Description / Properties                                                                                           | Computational Cost                               | Problem Domain Pros / Cons                                                                                                                                       |
|-----------------|--------------------------------------------------------------------------------------------------------------------|--------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| None (identity) | No projection; byte-for-byte equivalent to un-regioned optimizer.                                                  | Zero overhead                                    | **Pro:** no cost when unconstrained. **Con:** n/a.                                                                                                               |
| Orthant         | OWL-QN style; clamps coordinates crossing zero to the current sign's orthant. Elementwise.                         | `O(n)` elementwise clamp; `vmap`/`jit`-able      | **Pro:** encourages sparsity; with L-BFGS reproduces OWL-QN. **Con:** only relevant for L1/sparsity objectives.                                                  |
| Trust-Region    | Radially clips step to `‖x_new − x‖₂ ≤ Δ`; optional adaptive radius via `ρ = ared/pred` (exact along-path model).  | `O(n)` clip; adaptive radius bookkeeping         | **Pro:** stabilizes aggressive oracle steps; principled predicted-reduction model. **Con:** chord/arc mismatch on curved path requires gentle shrink heuristics. |
| No-Decrease     | Multi-objective guard; projects step onto `{s : ⟨∇g, s⟩ ≤ 0}` to not increase a secondary objective `g`.           | One extra gradient of `g` per projection; `O(n)` | **Pro:** continual learning / constrained fine-tuning without harming `g`. **Con:** extra `g`-gradient cost; opt-in only.                                        |
| Box             | Elementwise bounds `lo ≤ x_new ≤ hi` via `clip`; either side may be `None` (±inf).                                 | `O(n)` clip                                      | **Pro:** trivial, cheap box constraints. **Con:** axis-aligned only.                                                                                             |
| Quantization    | Confines each weight to its starting value's rounding cell; draws toward representable grid without hard-snapping. | `O(n)` per projection                            | **Pro:** geometric quantization-delta regularizer for low-precision deployment. **Con:** specialized to quantization use cases.                                  |

### 4.1 Region Combinators

| Name         | Description / Properties                                                      | Computational Cost            | Problem Domain Pros / Cons                                                                                       |
|--------------|-------------------------------------------------------------------------------|-------------------------------|------------------------------------------------------------------------------------------------------------------|
| Sequential   | Applies child projections in order (e.g. box ∩ trust-region).                 | Sum of child projection costs | **Pro:** simple composition of constraints. **Con:** ordering may not give exact intersection.                   |
| Intersection | Dykstra-style alternating projection toward the true intersection of regions. | Iterated projections          | **Pro:** converges to genuine intersection of convex constraints. **Con:** iterative — costlier than Sequential. |
